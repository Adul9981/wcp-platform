# 平台建设原则（不可违背的标准）

> 这是宪法级文件。任何功能、任何代码、任何取舍若与此冲突，以本文件为准。
> 新增原则需谨慎，删除原则需说明理由并记录。

---

## P0 · 数据真实性（最高优先级）
1. **不臆造数据**。所有展示的赔率/概率/成交量必须来自真实 API 抓取，标注 `snapshot_ts`。
2. **数据有时效标记**。任何页面展示的价格必须显示"数据时间"，过期数据明确标灰/警示。
3. **来源可追溯**。每条选题保留其 Polymarket 原始 id 和原始链接，可一键跳转核对。
4. **失败要显性**。抓取失败、API 超时、字段缺失 → 页面如实显示"数据不可用"，绝不用旧值假装新值。

## P1 · 资金安全
1. **读写分离**。行情读取（无需认证）与下单交易（需私钥）严格分离为两套系统。
2. **私钥永不入库/入日志/入前端**。仅在本地隔离环境，环境变量注入。
3. **默认纸面交易**。任何真实下单功能默认关闭，需显式开启 + 二次确认。
4. **仓位由用户自定**。仓位管理系统已取消，平台不做自动评分门槛/单注上限。仅呈现机会与盈亏比，下注多少由用户决定。

## P2 · 决策可信
1. **策略可解释**。每个推荐必须能说清"为什么"（命中哪条策略、实力/分组/阶段依据），不做黑箱推荐。**不声称"公平价值/edge"**——未接外部赔率，不博市场更准（[不追信息差](../3_策略分析库/核心策略清单.md)）。
2. **不夸大胜率**。展示历史命中率必须包含样本量与时间范围，不挑好看的区间。
3. **盈亏如实记录**。复盘库记录所有注单（含未操作/失败），不只记录盈利。

## P3 · 工程标准
1. **先规划后建设**。每个模块先进规划/待办库，评审后再写代码。
2. **小步可回滚**。每次变更聚焦单一目标，保留回退能力。
3. **配置与密钥分离**。密钥走环境变量/密钥管理，配置走配置文件，代码不硬编码。
4. **接口带超时与重试**。所有外部请求设超时、带退避重试、带 User-Agent（urllib直连会403）。
5. **尊重数据源**。控制请求频率，避免高频轰炸 Polymarket；优先批量端点(`/books`/`/prices` 最多500)。

## P4 · 产品体验
1. **三榜恒在**：📅 今日关注 / 🛡️ 低风险事件 / 🎲 高风险事件 —— 是平台第一屏的核心，不被次要功能挤占。（旧名：最值得关注/最被忽视/最大盈亏比，代码迁移见待办库）
2. **低流动不剔除**。新上线/低成交选题要有专门入口，不能因成交量小被埋没；低流动是正常状态，不是差信号。
3. **移动可读**。核心信息在手机上一屏可读。
4. **中文优先**。界面、文案、复盘默认中文。

## P5 · 合规与边界
1. **明确风险声明**。涉及下注的内容附风险提示，不构成投资建议。
2. **遵守地域法规**。发布/带单功能上线前确认目标地区合规性。
3. **不诱导**。文案系统不使用"稳赚""必中"等误导性表述（见文案禁忌）。

## P6 · 产品形态（已定位：公开付费产品）
1. **只读 + 决策辅助**。平台不持有用户资金、不代下单。所有交易动作由用户在 Polymarket 自行完成。
2. **跳转必带邀请码**。所有跳转 Polymarket 的链接统一附带运营方邀请/推荐码（Builder 归因），由配置集中管理，禁止散落硬编码。
3. **邀请码透明**。跳转处如实告知含推荐关系，不隐瞒（合规 + 信任）。
4. **多用户就绪的架构**。即使 MVP 单用户，数据/服务分层也要为将来多用户、订阅付费留出扩展位（用户态与公共数据态分离）。
5. **订阅付费后置**。先保证核心功能(数据管道→三榜→展示)达标，付费墙最后接。
